## How a UX Content Designer Thinks

Great UX writing doesn't start with writing. It starts with understanding. Follow this process before generating any content.

### Phase 1: Understand the Context

Before writing a single word, answer these questions:

**About the user:**
- Who is this person right now? (New user? Power user? Admin? Someone in a crisis?)
- What were they doing 5 seconds before they saw this screen?
- What do they need to do next?
- What's their emotional state? (Frustrated? Curious? In a hurry? Anxious?)
- How technically sophisticated are they?

**About the moment:**
- Is this a high-stakes moment (payment, deletion, error) or low-stakes (settings, preferences)?
- Is the user in flow (completing a task) or exploring (browsing, learning)?
- Will they read carefully or glance and move on?
- Are they on mobile (one thumb, distracted) or desktop (focused, mouse)?

**About the business:**
- What action does the business want the user to take?
- What happens if the user does nothing?
- Is there a conversion goal, retention goal, or support-reduction goal?

### Phase 2: Question the Brief

Never accept a content request at face value. Real designers push back:

- **"Welcome screen"** → For whom? First-time users or returning users? These need completely different content.
- **"Error message"** → Is this blocking the user or just informational? Can the user fix it themselves?
- **"CTA button"** → Primary or secondary action? What's the commitment level? (signing up vs. buying vs. just exploring)
- **"Notification"** → Is this worth interrupting someone's day? What happens if they miss it?
- **"Empty state"** → First-time empty (educational) or cleared-by-user empty (celebratory) or no-results empty (helpful)?

The content you write depends entirely on the answers to these questions. Different answers produce fundamentally different content.

### Phase 3: Map the Constraints

Every piece of UI text exists within constraints. Identify them:

- **Space:** How many characters fit? Does it need to work on mobile AND desktop?
- **Platform:** iOS has different conventions than Android. Web is different from native apps.
- **Localization:** Will this be translated? Some languages expand 30-40%.
- **Accessibility:** Screen readers announce this text — does it make sense spoken aloud?
- **Consistency:** What terms does the product already use? ("Save" or "Submit"? "Workspace" or "Project"?)
- **Legal:** Are there compliance requirements (GDPR consent, financial disclosures)?

### Phase 4: Generate with Intention

Now write — but with discipline:

- **Lead with the user's need**, not the system's state
- **Front-load the key information** — mobile users read the first 3-5 words
- **One idea per sentence** — don't stack multiple concepts
- **Use the user's vocabulary**, not internal jargon
- **Write 3-5 variations** with different approaches (emotional, logical, minimal, action-oriented)
- **Show your reasoning** — explain WHY each variation works, not just WHAT it says

### Phase 5: Validate Against Reality

Before delivering, stress-test:

- **Read it aloud.** Does it sound natural? Would a human say this?
- **Truncation test.** What happens if only the first 40 characters display?
- **Translation test.** Will this survive a 30% expansion into German?
- **Screen reader test.** Does this make sense without visual context?
- **Edge case test.** What if the user's name is 50 characters? What if there's no data?
- **Consistency test.** Does this match the tone and terminology used elsewhere in the product?
- **Emotional test.** If the user is frustrated (error) or anxious (payment), does this tone help or hurt?

## The Cardinal Rules

1. **Never assume — always ask.** Missing context produces generic content. Generic content fails.
2. **Constraints are creative fuel.** A 25-character button limit forces clarity. Embrace it.
3. **Write for the worst moment.** Design for the stressed, distracted, non-native-speaking user on a small phone in bright sunlight. If it works for them, it works for everyone.
4. **Every word must earn its place.** If removing a word doesn't change the meaning, remove it.
5. **Content is UI.** The words ARE the interface. They're not decoration on top of it.
